💡 **What:** Extracted the redundant `min`, `max`, and division (`// 2`) calculations for the velocity randomization bounds (`vel_rand`) outside of the `for` loops in `_generate_arpeggio_track` and `_generate_block_track`.

🎯 **Why:** `vel_rand` is a constant derived from user options. Recomputing its halved bounds (`-vel_rand // 2` and `max(1, vel_rand // 2)`) for every single note within a loop redundantly consumes CPU cycles, especially on longer progressions or fast arpeggios.

📊 **Measured Improvement:** Running a micro-benchmark to isolate the velocity generation block with 10 million iterations yielded an execution time decrease from ~16.99 seconds to ~13.80 seconds. This represents an ~18.7% performance improvement for this specific logic path. While the absolute time saved per run is small, this reduces unnecessary overhead in the MIDI generation loop.
